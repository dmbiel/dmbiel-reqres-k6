# Metrics Catalog

## Purpose

This catalog documents the metrics used by the k6 suite, how they map to test intent, and how they should be interpreted during CI review.

The goal is to make performance results reviewable without requiring every reviewer to inspect the k6 script internals.

## Metric Model

The project uses three metric layers:

| Layer | Source | Purpose |
|---|---|---|
| Built-in k6 metrics | k6 runtime | Overall HTTP, execution, and network behavior |
| Tagged HTTP metrics | request tags in test modules | Flow-level and endpoint-level filtering |
| Custom endpoint metrics | `helpers/metrics.js` | Endpoint-specific quality gates and reporting |

## Built-In Metrics

| Metric | Type | Used for |
|---|---|---|
| `http_req_failed` | Rate | Detects unexpected HTTP failures |
| `http_req_duration` | Trend | Measures end-to-end request duration |
| `checks` | Rate | Tracks functional validation pass rate |
| `http_reqs` | Counter | Shows total request volume |
| `iteration_duration` | Trend | Shows how long each scenario iteration takes |
| `vus` | Gauge | Shows active virtual users |
| `data_received` | Counter | Shows response payload volume |
| `data_sent` | Counter | Shows request payload volume |

## Request Tags

Each API flow can add request tags through the shared HTTP client.

| Tag | Example | Purpose |
|---|---|---|
| `environment` | `prod` | Separates results by target environment |
| `name` | `GET /api/users?page=1` | Gives readable request names in reports |
| `endpoint` | `list_users` | Filters endpoint-level duration metrics |
| `flow` | `standard`, `negative` | Groups similar behavior types |

Tagging makes the global metrics easier to explain. For example, the smoke scenario has separate duration thresholds for standard, negative, and delayed flows.

## Custom Endpoint Metrics

Custom metrics are defined in `helpers/metrics.js` and recorded from the test modules after each request.

| Endpoint metric | Type | Expected success condition |
|---|---|---|
| `endpoint_list_users_duration` | Trend | `GET /api/users?page=1` duration |
| `endpoint_list_users_success_rate` | Rate | status is `200` |
| `endpoint_single_user_duration` | Trend | `GET /api/users/2` duration |
| `endpoint_single_user_success_rate` | Rate | status is `200` |
| `endpoint_user_not_found_duration` | Trend | `GET /api/users/23` duration |
| `endpoint_user_not_found_success_rate` | Rate | status is `404` |
| `endpoint_create_user_duration` | Trend | `POST /api/users` duration |
| `endpoint_create_user_success_rate` | Rate | status is `201` |
| `endpoint_login_successful_duration` | Trend | `POST /api/login` duration |
| `endpoint_login_successful_success_rate` | Rate | status is `200` |
| `endpoint_delayed_users_duration` | Trend | `GET /api/users?delay=3` duration |
| `endpoint_delayed_users_success_rate` | Rate | status is `200` |

## Threshold Usage

Thresholds are intentionally defined at both global and endpoint levels.

| Threshold type | Example | Why it exists |
|---|---|---|
| Global error rate | `http_req_failed: rate<0.01` | Detects unexpected HTTP behavior across the run |
| Global duration | `http_req_duration: p(95)<1200` | Gives a simple scenario-level latency gate |
| Check rate | `checks: rate>0.95` | Validates response status and payload expectations |
| Endpoint success rate | `endpoint_list_users_success_rate: rate>0.95` | Identifies which flow is failing |
| Endpoint duration | `endpoint_list_users_duration: p(95)<1200` | Prevents a single endpoint regression from hiding in aggregate data |

## Reporting Usage

The GitHub Actions summary generated by `scripts/writeK6Summary.js` focuses on:

- scenario name;
- check pass rate;
- HTTP failure rate;
- total request count;
- p95, p99, and average duration;
- endpoint-level success and duration metrics when available.

The JSON summary artifact remains the source of truth for detailed investigation.

## Adding a New Metric

When adding a new API flow:

1. Add a stable endpoint key in `helpers/metrics.js`.
2. Record duration and success rate from the related test module.
3. Add request tags for `name`, `endpoint`, and `flow`.
4. Add scenario-specific thresholds only where the endpoint is executed.
5. Update this catalog and the quality gates documentation.

Do not add high-cardinality tags such as random IDs, timestamps, or dynamic payload values. They make metrics harder to compare and can produce noisy reports.
